# Тестовая модель

## 1. Цель
Этот документ описывает тестовую модель для автоматизации UI проекта для https://demowebshop.tricentis.com/ (в дальнейшем - веб приложение).
на базе Playwright и Javascript(в дальнейшем возможен переход на TypeScript). Цель — определить, какие части системы
тестируются, как тесты классифицируются и какие риски покрываются.

---

## 2. Система под тестированием
Система под тестированием — это веб-приложение e-commerce с основными
функциональными областями:
- Аутентификация
- Каталог товаров
- Корзина
- Оформление заказа (Checkout)

---

## 3. Уровни и типы тестирования

### 3.1 Smoke-тесты
Smoke-тесты проверяют, что приложение работает и базовый функционал
доступен после деплоя.

**Характеристики:**
- Быстрое выполнение
- Только happy path
- Допускается минимальная подготовка данных

**Покрытые области:**
- Открытие главной страницы
- Открытие страницы аутентификации пользователя
- Открытие страницы категории(при переходе из списка категорий и через кнопки в топ меню)
- Открытие страницы списка желаемого
- Открытие страницы корзины

---

### 3.2 Регрессионное тестирование
Регрессионные тесты проверяют, что существующий функционал работает
корректно после изменений в приложении.

**Характеристики:**
- Функциональные проверки
- Позитивные и негативные сценарии
- Изолированная проверка конкретных функций

**Покрытые области:**
- Логин (happy path)
- Логаут (happy path)
- Смена пароля
- Сортировка товаров по цене
- Фильтры
- Добавление товаров в корзину
- Изменение количества товара
- Удаление товаров из корзины
- Валидационные сообщения и ошибки


---

### 3.3 End-to-End (E2E) тесты
E2E-тесты проверяют полноценные пользовательские сценарии
от начала до конца.

**Характеристики:**
- Полное покрытие бизнес-процессов
- Минимальное количество тестов
- Реалистичное поведение пользователя

**Покрытые области:**
- Регистрация → Логин → Добавление в корзину → Оформление заказа → Подтверждение заказа

---

## 4. Стратегия тестовых данных
Тестовые данные генерируются динамически с помощью библиотеки Faker,
чтобы обеспечить уникальность и избежать конфликтов между прогоном тестов.

- Валидные пользователи для сценариев happy path
- Невалидные данные для негативных тестов
- Данные о товарах берутся из UI

---

## 5. Риски и приоритеты

| Область | Риск | Приоритет |
|---------|------|-----------|
| Аутентификация | Пользователь не может войти | Высокий |
| Корзина | Товар не добавляется | Высокий |
| Checkout | Заказ не создаётся | Высокий |
| Сортировка | Неверный порядок | Средний |
| Фильтры | Некорректные результаты | Средний |

---

## 6. Объем автоматизации
- Только UI автоматизация
- Браузеры: Chromium, Firefox, WebKit
- Среда: Тест / Demo

## 7. Что вне области тестирования
- Тестирование бэкенда и базы данных
- Интеграция с платежными провайдерами
- Производительность и нагрузочные тесты

---

## 8. Критерии начала и окончания

### Критерии начала
- Приложение развернуто
- Тестовая среда доступна
- Настроена генерация тестовых данных

### Критерии окончания
- Все smoke-тесты успешно пройдены
- Функциональные regression-тесты успешно пройдены
- Все e2e-тесты успешно пройдены
- Нет критических дефектов
- flaky тесты отсутствуют
